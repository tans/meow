# 创意喵 MVP 主交易链设计

## 1. 目标与范围

本设计聚焦创意喵第一阶段的主交易链 MVP，目标是在当前 monorepo 基础上，形成一条真正可实现、可验证、可扩展的核心闭环。

本次只覆盖以下链路：

- 商家发布公开任务
- 创作者浏览任务并投稿
- 商家审核投稿
- 商家发放基础奖、排名奖和额外打赏
- 任务截止后统一结算
- 创作者收益从冻结转为可提现
- 后台进行最小化任务巡检与账本查看

本次明确不包含：

- 真实充值通道
- 真实提现通道
- 完整申诉仲裁系统
- 完整信用体系
- AI 风控规则系统
- 完整版权治理

本次资金链路采用平台内账本模拟。

## 2. 核心产品决策

本设计基于以下已经确认的产品策略：

- 交易链路按端到端 MVP 设计，覆盖业务、API、用户端和最小后台入口
- 资金能力使用账本模拟，不接微信、支付宝、银行卡等真实支付通道
- 任务采用公开任务池，所有符合条件的创作者都可浏览和参与
- 任务采用轻审核机制，商家托管成功后可直接上线，平台仅支持事后下架
- 奖励体系第一版同时支持基础参与奖、排名奖和额外打赏
- 单任务投稿限制由商家在任务配置时设置，不引入信用分规则

## 3. 总体架构

本次实现采用单体业务流方案，不拆微服务。

### 3.1 应用边界

- `apps/wechat-miniapp`
  第一阶段主客户端，使用微信小程序原生框架实现
- `apps/admin`
  平台后台管理系统，承载任务巡检、账本查看和最小治理入口
- `apps/api`
  单体 API / BFF，承接主交易链全部业务编排
- `packages/database`
  数据库 schema、迁移、状态枚举与审计字段
- `packages/contracts`
  API 契约、状态字典、角色权限、页面流程定义
- `packages/domain-*`
  保留领域规则、流程名词和状态机规范

### 3.2 架构原则

- 任务状态与账本状态分离建模，通过业务动作联动
- 所有奖励先落到奖励明细和账本，再通过结算动作统一收敛
- 后台轻审核只影响任务对外可见性，不干预商家日常审稿行为
- 原生小程序、后续 iOS、Android 共享产品体验和接口契约，但不共享 UI 代码

## 4. 用户端设计

### 4.1 小程序产品壳

商家端和创作者端使用同一个小程序，不拆成两个产品。

产品形态定义为：

- 一个账号体系
- 一个小程序壳
- 两个工作台
- 一套共享的消息、钱包、任务详情和个人中心能力

用户可以在同一个账号下拥有 `merchant`、`creator` 两种角色中的一个或两个。

### 4.2 工作台切换

角色切换采用“同一小程序中的双工作台”设计，而不是“两个应用入口”设计。

建议做法：

- 首页顶部提供明确切换入口：`创作者工作台 / 商家工作台`
- 切换后改变底部导航和首页默认落点
- 消息、钱包、账户中心等共享页面不随角色切断

### 4.3 底部导航

创作者工作台：

- 任务池
- 我的投稿
- 收益
- 我的

商家工作台：

- 任务管理
- 发布任务
- 稿件审核
- 我的

### 4.4 页面组织

共享页面：

- 登录 / 注册
- 首页角色切换入口
- 任务详情页骨架
- 消息中心
- 钱包 / 收益概览
- 账户中心

商家专属页面：

- 发布任务
- 商家任务列表
- 商家任务详情管理视图
- 投稿审核页
- 评奖结算页

创作者专属页面：

- 公开任务池
- 我的投稿
- 投稿编辑页
- 收益明细页

### 4.5 原生端策略

当前优先实现微信小程序原生框架版本。

后续 iOS 与 Android 采用各自原生技术栈实现相同体验：

- iOS 建议 `SwiftUI`
- Android 建议 `Jetpack Compose`

共享对象不是 UI 代码，而是：

- 信息架构
- 页面流
- API 契约
- 状态机
- 文案规则
- 交互行为

## 5. 关键页面流

### 5.1 商家发任务流

页面流：

`商家工作台首页 -> 发布任务 -> 填写任务信息 -> 配置奖励与投稿上限 -> 确认预算托管 -> 发布成功 -> 任务详情`

约束：

- 发布任务必须是商家一级入口
- 托管预算确认必须发生在发布前最后一步
- 发布页需要明确展示预算拆分：基础奖、排名奖
- 打赏不属于发布时锁定预算，作为任务执行中的追加支出单独处理

### 5.2 商家审稿评奖流

页面流：

`任务管理 -> 任务详情 -> 稿件列表 -> 通过 / 驳回 -> 截止后评奖 -> 发起结算`

约束：

- 审稿与评奖分成两个动作阶段
- 任务进行中只做通过、驳回和额外打赏
- 排名奖必须在任务截止后统一处理

### 5.3 创作者参与流

页面流：

`任务池 -> 任务详情 -> 立即投稿 -> 上传作品 -> 提交待审核 -> 查看审核结果`

约束：

- 任务池是创作者默认首页
- 投稿入口必须在任务详情页中清晰可见
- 审核通过前允许修改或撤回投稿

### 5.4 创作者收益流

页面流：

`收益页 -> 冻结收益 -> 已解冻收益 -> 申请提现`

约束：

- 任务结算前收益仅显示为冻结中
- 任务结算后转为可提现余额
- 提现动作在第一版中只生成模拟出账记录

## 6. 数据模型

本次 MVP 保留 8 个核心实体。

### 6.1 User

表示登录主体，只负责账户身份。

### 6.2 UserRole

保存用户与角色关系，角色值为：

- `merchant`
- `creator`

同一个用户可以同时拥有两种角色。

### 6.3 Task

任务主体，包含：

- 商家
- 标题
- 描述
- 分类
- 截止时间
- 可见性
- 单人投稿上限
- 任务状态

### 6.4 TaskRewardPlan

保存任务奖励配置，包含：

- 基础奖金额
- 基础奖数量或上限
- 一等奖配置
- 二等奖配置
- 三等奖配置
- 优秀奖配置
- 奖励预算汇总

说明：

- `TaskRewardPlan` 只覆盖发布时需要锁定的基础奖和排名奖预算
- 额外打赏不属于初始奖励计划，按追加支出单独入账

### 6.5 Submission

创作者投稿实体，包含：

- 所属任务
- 投稿作者
- 素材内容
- 描述
- 审核状态
- 驳回原因

### 6.6 SubmissionReward

保存单个投稿获得的奖励明细，奖励类型包括：

- `base`
- `ranking`
- `tip`

### 6.7 LedgerAccount

平台内账本账户，至少包括：

- 商家余额账户
- 商家托管账户
- 创作者冻结收益账户
- 创作者可提现账户

### 6.8 LedgerEntry

账本流水，用于记录：

- 预算托管
- 奖励冻结
- 收益解冻
- 预算退回
- 提现模拟

## 7. 状态机设计

### 7.1 Task 状态

- `draft`
- `published`
- `paused`
- `ended`
- `settled`
- `closed`

说明：

- `draft`：草稿态，尚未发布
- `published`：已公开，可投稿
- `paused`：后台轻审核后临时下架
- `ended`：截止，不再接受投稿，等待评奖与结算
- `settled`：奖励与退款都已完成
- `closed`：归档态，可在第一版中作为内部状态保留

### 7.2 Submission 状态

- `submitted`
- `approved`
- `rejected`
- `withdrawn`

说明：

- `submitted`：待审核
- `approved`：审核通过
- `rejected`：驳回，必须记录原因
- `withdrawn`：创作者在待审期主动撤回

### 7.3 SubmissionReward 状态

- `frozen`
- `available`
- `cancelled`

说明：

- 所有奖励先进入 `frozen`
- 任务完成结算后转为 `available`
- 无效奖励进入 `cancelled`

## 8. 账本规则

### 8.1 商家账本动作

- 发布任务时，从商家余额账户转入商家托管账户
- 审核通过投稿时，不直接写入创作者可提现余额，只确认奖励资格
- 商家打赏时，从商家余额账户单笔划转到创作者冻结收益账户，不占用任务初始托管预算
- 任务结算时，将实际奖励金额从商家托管账户划转到创作者冻结账户
- 结算流程中，再把应解冻金额从创作者冻结账户转到创作者可提现账户
- 未使用预算从商家托管账户退回商家余额账户

### 8.2 创作者账本动作

- 任务结束前只能看到冻结收益
- 任务结算后冻结收益转为可提现收益
- 提现申请只允许从可提现余额扣减，并生成模拟出账流水

### 8.3 强约束

- 任务发布时必须一次性锁定基础奖和排名奖预算
- 基础奖资格来自投稿审核通过结果，而不是结算时重新推导
- 排名奖只能在任务截止后发放
- 打赏属于任务执行中的追加支出，不属于初始托管预算，但仍通过奖励明细纳入收益展示
- 后台轻审核只改变任务是否继续公开，不直接改动投稿审核状态

## 11. 错误处理与异常场景

MVP 至少需要对以下异常场景给出明确定义。

### 11.1 预算不足

商家发布任务时，如余额不足以覆盖总预算，必须阻断发布。

### 11.2 任务已下架或截止

创作者在提交投稿时必须二次校验任务状态；若任务已下架或截止，则拒绝投稿。

### 11.3 重复结算

已结算任务不可重复结算，接口需保证幂等。

### 11.4 审核冲突

同一投稿不可被重复审核；发生并发冲突时，后写入请求失败并提示刷新。

### 11.5 排名奖配置错误

若出现奖项数量超限、金额超预算、同一作品重复得奖，必须阻断评奖提交。

### 11.6 提现余额不足

提现只允许使用可提现余额，不允许动用冻结余额。

### 11.7 后台下架中的任务

任务被后台下架后，前台允许查看详情，但不能新增投稿。

### 11.8 上传成功但业务提交失败

素材上传成功但投稿创建失败时，前端必须明确提示“未保存成功”，后端需要保留可清理的临时文件引用策略。

## 12. 用户反馈原则

用户侧提示采用动作型文案，不暴露后台术语。

商家端示例：

- 预算不足，当前任务还不能发布
- 该任务已被平台暂时下架
- 该投稿已经处理过，请刷新列表

创作者端示例：

- 任务已截止，暂时不能投稿
- 你的作品还在审核中
- 当前可提现余额不足

后台端允许保留更清晰的状态说明和原因字段。

## 13. 测试策略

### 13.1 领域与服务层测试

重点验证：

- 任务状态流转
- 投稿审核状态流转
- 奖励计算
- 任务结算逻辑
- 账本流水生成

### 13.2 API 契约测试

重点验证：

- 发布任务
- 审核投稿
- 评奖
- 结算
- 提现模拟

### 13.3 Harness 场景测试

在现有 `apps/harness` 基础上，扩展为流程回放型校验，至少覆盖：

- 商家发布并托管任务
- 创作者投稿并通过审核
- 商家评奖并结算
- 创作者收益从冻结转可提现
- 后台下架任务后禁止新增投稿

## 14. 验收标准

本次 MVP 是否完成，以以下 5 条为准：

- 商家可以创建并发布公开任务
- 创作者可以浏览任务、投稿并查看审核结果
- 商家可以审核投稿、发基础奖、发排名奖、发打赏
- 任务结束后可以完成结算、退款和收益解冻
- 后台可以查看任务与账本，并下架异常任务

若以上 5 条成立，则本设计范围内的主交易链 MVP 视为闭环完成。
