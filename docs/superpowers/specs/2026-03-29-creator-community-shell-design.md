# 创意喵创作者社区化改版设计

## 1. 目标与范围

本设计聚焦微信小程序第一版的视觉气质与信息架构调整，目标是把当前偏工具后台的产品壳，改成更像互联网创作者社区的平台体验。

本次改版基于以下已确认方向：

- 整体气质参考“小红书感”，强调内容流、轻社交和社区氛围
- 页面结构采用“创作者社区 + 任务平台”混合形态
- 商家能力不再作为主导航心智出现
- 商家需求收进“我的”页，用单张弱化管理卡承载
- 主导航调整为：`悬赏大厅 / 获奖作品 / 我的`

本次改版只覆盖：

- 小程序一级信息架构调整
- 底部导航重组
- 三个主页面的内容模块重构
- 现有入口页向内容流页面的视觉转型
- 商家入口在“我的”页中的收口方式

本次明确不包含：

- 真实社区互动能力，例如关注、评论、点赞、私信
- 任务发布和审核流程本身的业务规则变更
- 后端数据结构大改
- 推荐算法、排序策略和运营后台建设
- 新的支付、提现、通知链路

## 2. 问题定义

当前微信小程序虽然已经完成 MVP 主链路页面骨架，但一级页面仍然明显偏“工具导航”：

- `工作台` 是摘要卡 + 数据列表
- `任务` 是跳转入口列表
- `收益` 独立成 tab，心智更接近工具后台
- `我的` 以身份切换为主体，不像个人主页

这种结构的问题在于：

- 用户进入后首先感受到的是管理动作，不是平台氛围
- 任务、作品、个人三类内容没有形成自然浏览关系
- 商家和创作者双角色都在一级层抢夺注意力，削弱了创作者社区感
- 页面虽然能导航，但缺少“可逛、可比较、可被吸引”的互联网平台体验

## 3. 核心产品决策

### 3.1 总体定位

本次改版后的产品壳定义为：

- 一个以创作者为默认心智的社区入口
- 一个仍然保留任务交易闭环的平台产品
- 一个把商家管理动作收敛到个人页下层的双角色小程序

### 3.2 默认心智

默认身份叙事从“我在管理任务”改成“我在看任务、看案例、经营我的创作者主页”。

因此，一级页面优先级调整为：

1. 发现任务
2. 观看优秀作品
3. 管理个人内容与收益

商家动作不再作为主导航出现，只在“我的”页通过低优先级入口触发。

### 3.3 设计原则

- 优先展示内容，再展示动作
- 优先建立社区感，再暴露管理能力
- 优先让用户浏览和理解平台，再让用户执行业务流程
- 保留现有 MVP 交易链路，但改变入口包装方式

## 4. 信息架构调整

### 4.1 底部导航

底部导航由四项改为三项：

- `悬赏大厅`
- `获奖作品`
- `我的`

对应收口如下：

- 原 `工作台` tab 取消，页面能力被重分配
- 原 `任务` 页升级为 `悬赏大厅`
- 原 `收益` tab 取消，收益能力并入 `我的`
- 原 `我的` 页从身份切换页升级为个人主页

### 4.2 页面职责

`悬赏大厅`

- 承担任务发现、频道浏览、任务比较和参与转化
- 取代原有工具型任务入口页

`获奖作品`

- 承担案例展示、平台榜单、品牌结果沉淀
- 用于建立平台真实感与优秀作品氛围

`我的`

- 承担个人身份、创作数据、投稿管理、收益查看和商家入口收纳

### 4.3 页面复用策略

为控制首版改造成本，优先采用现有页面骨架重命名和重构，而不是一次性铺开大量新页面。

建议复用方式：

- 现有 `pages/tasks/index` 重构为 `悬赏大厅`
- 现有 `pages/workspace/index` 重构为 `获奖作品`
- 现有 `pages/profile/index` 重构为新的 `我的`
- 现有 `pages/wallet/index` 从 tab 退出，保留为“我的”内二级入口

这种方案既能明显改变第一层体验，也能减少路由和测试面扩张。

## 5. 页面设计

### 5.1 悬赏大厅

#### 页面目标

把“任务中心”改造成具有平台氛围的任务内容流，让用户在首页就看到品牌任务、热门频道和明确转化按钮。

#### 页面结构

1. 顶部氛围区
2. 频道切换区
3. 任务内容流
4. 下拉刷新

#### 顶部氛围区

顶部不使用工作台式摘要卡，而是采用轻运营头图或品牌渐变卡片，承载一条平台化文案，例如：

- 今天有 28 个品牌任务在征稿
- 本周探店、穿搭类任务热度上升

该区域的职责是先建立活跃平台印象，而不是承载具体业务按钮。

#### 频道切换

顶部设置横向频道切换：

- `推荐`
- `品牌合作`
- `急单`
- `同城`

频道切换需要具备吸附感和明显状态变化，用于让页面更像互联网内容平台，而不是列表页。

#### 任务卡设计

任务卡采用内容卡片而不是列表行，字段顺序建议如下：

- 封面图或品牌视觉背景
- 任务标题
- 一句话任务要求
- 品牌名与任务标签
- 奖励信息：总预算、名次奖、参与奖
- 状态信息：距离截止时间、参与人数
- 底部动作：`查看详情` 与强化的 `立即报名`

#### 视觉重点

任务卡需要重点突出：

- 封面
- 预算
- 品牌归属
- 截止紧迫感

需要去掉当前“右侧一个进入胶囊”的工具感写法，改成整卡可点，主按钮只服务报名转化。

#### 运营标签

每张任务卡只显示一个高优先级标签，避免信息拥挤。标签候选包括：

- `平台精选`
- `奖金高`
- `即将截止`
- `新发布`

### 5.2 获奖作品

#### 页面目标

把“结果展示”做成“榜单 + 案例库”，让用户看到平台真实产出，建立对优秀作品的向往感和学习路径。

#### 页面结构

1. 顶部专题区
2. 榜单时间切换
3. 分类切换
4. 获奖作品流
5. 精选运营模块

#### 顶部专题区

顶部使用专题横幅承载阶段性主题，例如：

- 本周品牌精选
- 春季探店获奖合集
- 本月高转化短视频案例

该区域的职责是把“获奖作品”从冷公示页转成可浏览的内容区。

#### 榜单切换

页面提供两层轻切换：

- 时间维度：`本周 / 本月 / 全部`
- 分类维度：`全部 / 美妆 / 探店 / 穿搭 / 家居`

第一版可先使用本地假数据和静态切换状态，后续再接真实筛选。

#### 作品卡设计

作品卡相较任务卡更强调“封面”和“创作者”，字段顺序建议如下：

- 大封面
- 获奖标识：一等奖、优秀奖、品牌精选
- 作品标题或亮点描述
- 创作者头像与昵称
- 所属任务与品牌名
- 结果信息：获奖金额、品牌已采用等
- 底部动作：`看作品` 或 `查看案例`

#### 排序优先级

首版视觉排序建议为：

- 平台精选
- 一等奖作品
- 品牌已采用作品
- 其他获奖作品

这样能更快建立平台价值感。

### 5.3 我的

#### 页面目标

把“身份切换页”改造成“创作者个人页”，同时在页面下半区保留一张弱化的商家需求管理卡。

#### 页面结构

1. 顶部个人头部
2. 创作数据卡
3. 我的内容入口区
4. 商家合作卡

#### 顶部个人头部

第一屏做成创作者主页感，包含：

- 头像
- 昵称
- 简介
- 创作标签
- 一句近期状态文案

示例状态文案：

- 本周已投 6 个任务，入围 2 个

这个区域优先提供身份感，而不是业务管理入口。

#### 创作数据卡

头部下方放一张创作数据卡，展示：

- 累计获奖
- 本月投稿
- 品牌合作
- 预估收益

这使得收益能力自然并入“我的”，无需单独 tab。

#### 我的内容入口区

这里放创作者高频入口：

- 我的投稿
- 收益明细
- 草稿箱
- 合作记录

文案应偏创作者语境，而不是后台管理语境。

#### 商家合作卡

商家需求通过独立卡片收纳在页面下半区。

卡片名称推荐使用：

- `商家合作`

不建议直接使用“商家后台”或“任务管理中心”这类偏后台口径的名称。

卡片内部保留少量管理动作：

- 发布任务
- 审核稿件
- 结算管理

同时给出一条辅助文案：

- 需要发布品牌合作任务时，从这里进入商家侧管理

#### 身份切换策略

身份切换不再作为页面主体出现。

首版明确采用卡片内一个 `进入商家侧` 按钮，不额外提供顶部身份切换控件。

这样保留双角色能力，但不破坏页面的创作者社区气质。

## 6. 视觉方向

### 6.1 气质关键词

- 轻内容平台
- 真实品牌合作
- 卡片丰富
- 生活化但不松散
- 有平台感，不像后台

### 6.2 视觉策略

当前通用样式偏白底工具卡，改版后需要强化以下方向：

- 卡片层级更丰富，而不是所有卡都长得一样
- 顶部区域允许使用更强的背景、渐变和图形氛围
- 封面图容器成为关键视觉元素
- 重点按钮强调“参与”和“查看案例”，弱化“进入”
- 标签、频道和专题模块要更像内容平台的运营位

### 6.3 风格约束

虽然参考“小红书感”，但不能直接复制红白模板。需要保留“品牌任务平台”的交易属性，避免做成纯生活社区。

## 7. 数据与实现边界

### 7.1 数据来源策略

第一版以本地 view-model 假数据为主，优先完成页面气质和模块结构调整。

需要补充的数据类型包括：

- 悬赏大厅频道配置
- 任务卡封面、标签、预算、参与人数、截止信息
- 获奖作品专题、榜单状态、作品封面、作者信息、获奖信息
- 个人页标签、状态文案、创作数据、常用入口、商家合作卡动作

### 7.2 对现有代码的影响范围

重点影响以下区域：

- `miniprogram/app.json` 的 tabBar 和页面文案
- `pages/tasks/index.*`
- `pages/workspace/index.*`
- `pages/profile/index.*`
- `pages/wallet/index.*` 的入口方式
- `src/services/workspace.js` 中的页面 view-model 组织方式

### 7.3 不做的事情

本轮不改动：

- 商家发布、审核、结算等二级业务页面的表单流程
- 创作者投稿详情页的业务逻辑
- 后端 API
- 角色存储机制本身

## 8. 交互与状态

### 8.1 页面交互

首版需要支持的关键交互：

- `悬赏大厅` 频道切换
- 任务卡点击进入详情
- 任务卡主按钮进入报名链路
- `获奖作品` 的榜单与分类切换
- `我的` 页内容入口跳转
- 商家合作卡进入商家管理能力

### 8.2 空状态

为了避免页面显得像未完成原型，空状态也应采用社区语境文案。

例如：

- 悬赏大厅空状态：`新的品牌合作正在路上`
- 获奖作品空状态：`下一批优秀作品正在评选中`
- 我的投稿空状态：`先去悬赏大厅挑一个喜欢的任务`

### 8.3 弱化后台感的交互约束

以下表达应尽量减少：

- 管理中心
- 工作台
- 进入
- 当前身份

优先使用：

- 立即报名
- 看作品
- 商家合作
- 合作记录

## 9. 测试与验证

### 9.1 验证目标

改版验收重点不是业务是否新增，而是页面是否从工具导航成功转成社区化平台外观。

### 9.2 最小验证范围

- tabBar 是否成功切换为三个主入口
- `悬赏大厅` 是否呈现内容流而不是功能列表
- `获奖作品` 是否具备榜单和案例感
- `我的` 是否首先像个人页，其次才是管理入口
- 商家能力是否被成功收纳进“我的”页卡片
- 原有二级业务路径是否仍然可达

### 9.3 自动化与手动验证

自动化测试重点：

- view-model 输出结构变化
- 页面关键文案和字段映射
- tabBar 配置调整后的基础稳定性

手动验证重点：

- 真机或开发者工具中页面第一眼气质
- 卡片层级、封面占比、按钮优先级
- 商家卡片是否弱化得足够自然

## 10. 实施建议

建议按以下顺序实施：

1. 先改 tabBar 与页面定位
2. 再重构三个一级页面的 view-model
3. 然后改 WXML 结构
4. 最后统一升级公共样式，去掉工具页气质

实施过程中应优先保证一级页面体验完成度，再处理细节动画和次要视觉修饰。
