# 创意喵 Web 移动壳与小程序一致化设计

## 1. 目标与范围

本设计聚焦 `apps/web` 的下一阶段落地：在保留 Web 登录流程的前提下，将登录后的用户端体验改造成“浏览器里的小程序”，以移动单栏布局完整补齐当前微信小程序的主流程页面，并将整体视觉切换为蓝白风格。

本次设计基于以下已确认约束：

- Web 登录是额外流程，登录后业务交互全部按小程序对齐
- Web 不做桌面工作台，不做双栏后台化适配
- 登录后的业务页保持移动布局和小程序相同的页面顺序
- 创作者与商家流程都要补齐到 Web
- 如发现可优化点，只记录，不混入本次实现

本次覆盖：

- Web 登录后移动壳重构
- Web 路由与小程序页面一一映射
- 创作者主线页面补齐
- 商家主线页面补齐
- 蓝白视觉系统落地
- 浏览器环境下的 URL、返回、刷新恢复适配

本次不包含：

- 小程序页面结构重做
- Admin 改版
- 服务端业务规则扩展
- 额外的新交互能力，例如批量审核、拖拽、复杂筛选
- 任何脱离小程序现状的体验优化直接落地

## 2. 当前现状与问题

当前仓库中：

- `apps/wechat-miniapp`
  已具备创作者 tab 结构，以及商家发布、审核、结算等主链页面
- `apps/web`
  仅完成了登录页、角色切换和少量列表型页面

当前 Web 与小程序存在四类偏差：

1. 页面覆盖不足
   - Web 缺少任务详情、投稿编辑、收益、资料页、商家发布、审核、结算等完整页面
2. 信息架构不一致
   - Web 当前是“工作台壳 + 页面块”，小程序是“tab + 二级页”结构
3. 交互节奏不一致
   - 小程序大量使用“点卡片进详情、提交后返回上一页、二级页原地反馈”的流程
   - Web 当前更像桌面概览页
4. 视觉风格不一致
   - Web 已改出的版本是暖色编辑部风格
   - 本次已确认要切成蓝白风格

因此本轮工作的核心不是简单换色，而是把 Web 业务体验与小程序重新拉齐。

## 3. 核心产品决策

### 3.1 产品形态

Web 定义为：

- 一个带登录入口的浏览器用户端
- 登录后进入移动单栏业务壳
- 业务内容、页面顺序、文案、动作和跳转节奏尽量与小程序保持一致

### 3.2 一致性原则

Web 与小程序的一致性目标包括：

- 一级入口一致
- 二级页面一致
- 主按钮文案一致
- 筛选与切换方式一致
- 跳转顺序一致
- 操作后的停留页面与反馈方式一致

Web 允许额外增加的能力仅限于浏览器必需能力：

- URL 路由
- 浏览器返回
- 刷新后恢复当前页面
- 登录态恢复

### 3.3 本轮不做的优化

以下内容只做记录，不进入本轮实现：

- 商家发布成功后自动强跳任务详情
- 收益页增加更强的金额层级设计
- 审核页批量动作
- 更复杂的 loading skeleton
- 更强的空状态插画

## 4. 总体结构

Web 拆分为两层：

### 4.1 外层：登录层

- 路径：`/login`
- 负责账号登录、会话恢复、进入业务壳
- 这是 Web 独有层，不要求与小程序一致
- 视觉切到同一蓝白体系

### 4.2 内层：移动业务壳

- 登录后进入固定移动宽度容器
- 内容采用小程序风格的单栏堆叠布局
- 创作者主线保留底部 tab
- 非 tab 页显示顶栏返回
- 商家页从“我的”中的商家入口进入，不额外增加 Web 独有商家主页

## 5. 路由与页面映射

### 5.1 创作者主线

- `/tasks`
  对齐 `pages/tasks/index`
- `/workspace`
  对齐 `pages/workspace/index`
- `/profile`
  对齐 `pages/profile/index`
- `/wallet`
  对齐 `pages/wallet/index`
- `/creator/task-feed`
  对齐 `pages/creator/task-feed/index`
- `/creator/task/:taskId`
  对齐 `pages/creator/task-detail/index`
- `/creator/submission-edit`
  对齐 `pages/creator/submission-edit/index`
- `/creator/submission-edit/:submissionId`
  对齐 `pages/creator/submission-edit/index?submissionId=...`
- `/creator/earnings`
  对齐 `pages/creator/earnings/index`

### 5.2 商家主线

- `/merchant/task-create`
  对齐 `pages/merchant/task-create/index`
- `/merchant/task/:taskId`
  对齐 `pages/merchant/task-detail/index`
- `/merchant/review/:taskId`
  对齐 `pages/merchant/review/index`
- `/merchant/settlement/:taskId`
  对齐 `pages/merchant/settlement/index`

### 5.3 路由规则

- 未登录统一进入 `/login`
- 已登录且访问根路径时：
  - 创作者默认进入 `/tasks`
  - 商家默认进入 `/merchant/task-create`
- 角色切换后，如果当前路由不属于目标角色，则跳转到目标角色默认入口
- URL 参数优先于内存态
- 缺少必须参数时，才允许回退到“最近选中任务”兼容逻辑

## 6. 页面交互设计

### 6.1 悬赏大厅

结构严格对齐小程序：

1. 顶部 hero
2. 横向频道筛选条
3. 任务卡列表

交互：

- 频道项为 `推荐 / 品牌合作 / 急单 / 同城`
- 点击频道原地切列表，不跳页
- 点击任务卡进入 `任务详情`

### 6.2 获奖作品

结构严格对齐小程序：

1. 顶部专题 banner
2. 时间筛选条
3. 分类筛选条
4. 案例卡列表

交互：

- 时间项为 `本周 / 本月 / 全部`
- 分类项为 `全部 / 美妆 / 探店 / 穿搭 / 家居`
- 所有筛选原地更新列表，不跳页

### 6.3 我的

结构严格对齐小程序：

1. 个人信息卡
2. 数据统计卡
3. 快捷入口列表
4. 商家入口卡

快捷入口保持小程序顺序：

- 我的投稿
- 收益明细
- 草稿箱
- 合作记录

交互：

- 点击快捷入口进入对应二级页
- 点击商家入口先切角色，再进入 `/merchant/task-create`

### 6.4 创作者任务详情

结构：

1. 页面标题卡
2. 任务信息区
3. 我的投稿列表
4. 反馈 / 错误区

交互：

- 可投稿时显示主按钮 `立即投稿`
- 点击进入 `/creator/submission-edit`
- 投稿列表中的 `修改` 进入 `/creator/submission-edit/:submissionId`
- `撤回` 为原地操作，成功后刷新当前页并显示反馈

### 6.5 创作者投稿编辑

结构：

1. 页面标题卡
2. 投稿表单
3. 结果 / 错误区

字段顺序与小程序一致：

- 素材链接
- 作品描述

交互：

- 新建状态按钮为 `提交作品`
- 编辑状态按钮为 `保存修改`
- 提交成功后返回上一页

### 6.6 创作者任务池

结构：

1. 页面标题卡
2. 任务列表

交互：

- 点击任务行进入 `/creator/task/:taskId`

### 6.7 收益明细与收益页

`/wallet` 与 `/creator/earnings` 都按小程序现状处理：

- 页面顶部保留标题与摘要
- 列表项采用简单行结构
- 仅展示标签和值，不额外增加复杂图表

### 6.8 商家发布任务

字段顺序与小程序一致：

- 任务标题
- 基础奖金额
- 基础奖份数
- 排名奖总额

交互：

- 预算汇总实时更新
- 主按钮为 `创建并发布任务`
- 发布成功后默认停留当前页，并展示结果反馈

### 6.9 商家任务详情

展示内容按小程序顺序：

- 标题
- 状态
- 奖励
- 投稿数
- 锁定预算
- 已发奖励

动作区：

- `前往审核`
- `前往结算`

### 6.10 商家审核

页面按当前任务的投稿列表展开。

每个投稿项根据状态显示动作：

- `通过`
- `打赏`
- `排名奖`

交互：

- 所有动作都在当前页完成
- 动作成功后刷新列表并显示反馈

### 6.11 商家结算

结构：

1. 页面标题卡
2. 结算摘要
3. 执行结算按钮
4. 结果 / 错误区

交互：

- 执行结算后停留本页
- 刷新结算摘要与奖励预览
- 展示 `任务 xxx 已结算` 类反馈

## 7. 视觉设计

### 7.1 视觉方向

登录后的业务页统一采用蓝白风格：

- 页面背景：冷白到浅蓝的渐变
- 容器卡片：纯白
- 主强调色：中亮蓝
- 次强调色：浅蓝、雾蓝边框与浅底
- 成功：少量绿色
- 错误：少量红色

### 7.2 布局原则

- 固定移动宽度容器
- 单栏堆叠
- 大圆角
- 轻阴影
- 保留小程序的轻量空白与信息密度

明确禁止：

- 桌面双栏
- 侧边导航
- Dashboard 式统计大盘
- 与小程序无关的海报化运营头图
- 过重的拟物阴影与复杂装饰

### 7.3 组件约束

统一组件策略如下：

- `Hero / Banner`
  保持小程序结构，只切蓝白配色
- `Chip`
  默认浅蓝底，激活态蓝底白字
- `Card / List Row`
  保持白底和单栏结构，不改字段顺序
- `Primary Button`
  蓝底白字
- `Ghost Button`
  白底蓝边蓝字
- `Pill`
  用于状态与反馈，不承担主要布局

## 8. 浏览器适配规则

浏览器适配只做必要增强：

- 使用 URL 承载当前页与任务参数
- 刷新后可恢复当前页面
- 非 tab 页提供返回能力
- 会话恢复失败时统一回登录页
- 底部 tab 固定在移动容器底部

浏览器适配不改变业务语义：

- 不增加桌面侧栏
- 不增加 hover 专属流程
- 不增加独立的工作台首页

## 9. 数据与状态约束

- 页面读模型优先复用现有 API 契约
- 任务 ID、投稿 ID 从 URL 读取
- 页面内筛选状态仅保存在当前路由页面内部
- 角色切换后，重新拉取目标角色相关数据
- 失败反馈文案尽量沿用小程序现有表达
