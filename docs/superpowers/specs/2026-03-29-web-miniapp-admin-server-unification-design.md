# 创意喵 Web / 小程序 / Admin / Server 一体化首版设计

## 1. 目标与范围

本设计聚焦创意喵当前 monorepo 的下一阶段落地：在已存在的小程序基础上，加速补齐 Web 用户端、平台运营后台和统一服务端，使三端共享同一套账号、角色、任务主链和账务数据。

本次设计的目标不是重新定义产品，而是把现有两份已确认设计收束成一个可执行的一体化方案：

- 延续 `2026-03-28` 的 MVP 主交易链
- 延续 `2026-03-29` 的创作者社区壳方向
- 将 Web、Mini Program、Admin 和 API 统一到真实后端和真实数据

本次覆盖：

- 用户 Web 应用首版
- 微信小程序接入统一认证与真实 API
- 平台运营后台从 demo 数据升级为真实后台
- 服务端统一认证、角色切换、任务主链、账务读写和后台治理动作
- 统一数据库模型和三端契约

本次不包含：

- 真实短信、微信扫码、OAuth 登录
- 申诉与裁决系统
- 复杂 RBAC 和多级权限平台
- 完整内容治理体系
- 推荐算法、社交互动和运营配置中心

## 2. 当前现状与问题

当前仓库已经具备以下基础：

- `apps/wechat-miniapp`
  已有创作者社区化小程序壳和商家/创作者主链页面
- `apps/admin`
  已有运营后台视觉壳，但数据来源仍是本地 preview 数据
- `apps/api`
  已有基于 `merchant/*`、`creator/*` 的 MVP 主链接口
- `packages/database`
  已有数据库包和测试基础

当前阻塞点主要有四类：

1. 小程序仍依赖 demo session 语义，不能支撑真实三端统一登录
2. Web 用户端尚未真正落地，`apps/app` 仍只是 blueprint
3. Admin 尚未接入真实后端，不具备运营闭环动作
4. API 缺少统一认证、角色切换、operator 接口和操作审计

因此本轮工作的核心不是单做一个前端，而是把“同一业务真相”建立起来。

## 3. 核心产品决策

本设计基于以下确认结论：

- Web 与小程序并行同步，不做“一个替代另一个”的路线
- Web 与小程序能力同构、体验同构，但 Web 允许桌面化自适应布局
- 小程序不推翻重做，直接在现有页面基础上切到统一后端
- Web 采用“一个应用 + 登录 + 角色切换”模式
- 登录后默认采用社区优先壳，不先进入业务系统式工作台分流
- 商家和创作者都使用 Web 与小程序
- Admin 只给平台运营使用，不给商家作为工作台
- 首版登录采用演示型账号体系，但必须是“真实会话”，不是前端假状态
- 后端首版采用真实数据库和统一 API，不接受继续用大面积 mock 过渡
- 风控/申诉本轮延后，不纳入当前一体化交付闭环

## 4. 总体架构

本次实现采用“单体后端 + 多前端”的统一方案，不拆微服务。

### 4.1 应用边界

- `apps/web`
  新建正式用户 Web 应用，承载商家与创作者双角色体验
- `apps/wechat-miniapp`
  保留现有原生小程序，继续作为正式用户端之一
- `apps/admin`
  平台运营后台，负责任务治理、用户治理、账务巡检和高风险动作
- `apps/api`
  三端统一后端，承接认证、会话、角色、任务、投稿、奖励、结算、账务和后台治理
- `packages/database`
  持久化模型、schema、测试库、seed 和审计字段
- `packages/contracts`
  路由契约、角色矩阵、端侧读模型和状态字典

### 4.2 架构原则

- 业务真相只存在于 `apps/api + packages/database`
- Web、小程序、Admin 都只消费 API，不在客户端持有工作流规则
- 角色切换由会话层负责，不由前端硬编码用户身份
- 任务状态、奖励状态、账务状态和治理状态分离建模
- 所有 operator 高风险动作必须进入审计日志

## 5. 交付阶段

为了兼顾速度和稳定性，本次统一设计拆成三个实现阶段，但属于同一个首版交付。

### 5.1 Phase 1: Unified Server Foundation

目标：

- 建立真实登录、会话和角色切换
- 将任务、投稿、钱包、结算主链全部接到真实数据库
- 提供 operator admin 的基础查询与治理接口

产出：

- `auth/*` 接口
- `admin/*` 接口
- 真正的 session 机制
- 用户、角色、会话、审计表

### 5.2 Phase 2: User Surfaces Sync

目标：

- Web 首版上线
- 小程序切到统一后端
- 商家与创作者在两端都可完成核心主链

产出：

- Web 登录壳和双角色导航
- Web 创作者主链
- Web 商家主链
- 小程序登录态与数据层切换

### 5.3 Phase 3: Operator Closure

目标：

- Admin 从静态后台升级为可执行运营台

产出：

- Dashboard 真实数据
- 任务治理动作
- 用户治理动作
- 资金巡检视图
- 操作审计记录

## 6. 用户端设计

### 6.1 Web 形态

Web 采用单一应用壳，登录后依据当前激活角色展示对应导航。

默认进入逻辑：

- 未登录 -> 登录页
- 已登录且激活角色为 `creator` -> 进入社区优先壳
- 已登录且激活角色为 `merchant` -> 进入商家工作台壳
- 双角色用户可在页面顶部显式切换角色

### 6.2 Web 创作者信息架构

创作者默认导航：

- `悬赏大厅`
- `获奖作品`
- `我的`

创作者核心页面：

- 任务流
- 任务详情
- 投稿编辑
- 收益明细
- 个人主页

桌面适配原则：

- 首页可采用双栏或三栏布局承载更多信息
- 列表页可加入侧栏筛选和固定摘要
- 详情页可将信息块展开为更完整的模块化布局
- 视觉心智与小程序一致，但不追求逐像素复刻

### 6.3 Web 商家信息架构

商家导航：

- `任务管理`
- `发布任务`
- `稿件审核`
- `我的`

商家核心页面：

- 商家任务列表
- 任务详情管理视图
- 发布任务
- 稿件审核
- 结算确认
- 商家钱包与账户摘要

### 6.4 小程序策略

小程序本轮不做大改版，原则如下：

- 保留当前页面组织与社区化导航
- 保留商家/创作者页面职责
- 必要时只新增轻量登录初始化页或启动态恢复逻辑
- 将数据请求、登录态和角色态切换到统一 API

即：

- 小程序继续是正式端
- 但不再拥有独立 demo 账户语义

## 7. Admin 设计

Admin 明确为 operator-only 后台。

首版菜单建议固定为：

- `系统总览`
- `任务管理`
- `用户管理`
- `资金管理`

### 7.1 Dashboard

展示：

- 待治理任务数
- 活跃商家与创作者摘要
- 资金异常提醒
- 最近 operator 操作记录

### 7.2 任务管理

支持：

- 按状态浏览任务
- 查看任务详情、奖励配置、投稿摘要、账务摘要
- 执行任务下架
- 执行任务恢复

### 7.3 用户管理

支持：

- 浏览商家/创作者账号
- 查看角色、状态、最近行为摘要
- 执行封禁
- 执行解禁

### 7.4 资金管理

支持：

- 查看预算托管、冻结收益、退款流水
- 对异常账务记录打标
- 浏览账务相关 operator 操作

本轮明确不做：

- 复杂权限系统
- 完整配置中心
- 审批流引擎
- 申诉/内容治理模块

## 8. 服务端设计

### 8.1 接口层结构

当前 `apps/api` 在现有 `creator/*` 与 `merchant/*` 基础上扩展为：

- `auth/*`
- `creator/*`
- `merchant/*`
- `admin/*`

其中：

- `auth/*` 管理登录、退出、当前会话和角色切换
- `creator/*` 管理创作者主链
- `merchant/*` 管理商家主链
- `admin/*` 管理 operator 查询与治理动作

### 8.2 登录与会话

首版登录采用演示型账号体系，但必须是真实后端会话。

方案：

- 数据库中 seed 演示账号与角色
- 登录页提交手机号或邮箱样式账号 + 演示验证码或密码
- 后端校验后签发 session
- Web 使用 cookie 或 bearer session
- 小程序使用 token 或会话标识存储
- `GET /auth/session` 返回当前用户、角色列表、激活角色、基础个人摘要

### 8.3 角色切换

角色切换不重新登录。

接口：

- `POST /auth/switch-role`

行为：

- 校验当前用户是否拥有目标角色
- 更新当前会话中的激活角色
- 返回新角色下的导航摘要和必要用户信息

### 8.4 用户主链

创作者接口继续承载：

- 浏览公开任务
- 查看任务详情
- 创建投稿
- 修改投稿
- 撤回投稿
- 查看收益

商家接口继续承载：

- 创建任务草稿
- 发布任务
- 查看任务列表
- 查看任务详情
- 审核投稿
- 发放打赏
- 分配排名奖
- 发起结算
- 查看商家钱包

### 8.5 Admin 接口

新增 operator 接口至少包含：

- dashboard snapshot
- task list / task detail
- user list / user detail
- ledger list / ledger detail
- pause task
- resume task
- ban user
- unban user
- mark ledger anomaly

### 8.6 审计

以下动作必须记录到 `operator_action_log`：

- 任务下架
- 任务恢复
- 用户封禁
- 用户解禁
- 账务异常标记

记录字段至少包括：

- operator 用户 ID
- 动作类型
- 目标类型
- 目标 ID
- 原因
- 时间
- 扩展 payload

## 9. 数据模型

在现有任务主链实体基础上，补齐以下核心实体。

### 9.1 `user`

登录主体，保存基础账户资料和状态。

### 9.2 `user_role`

用户与角色关系表。

角色值首版限定为：

- `creator`
- `merchant`
- `operator`

### 9.3 `auth_session`

保存当前会话、激活角色、过期时间和客户端类型。

### 9.4 `operator_action_log`

保存后台治理动作审计记录。

### 9.5 既有主链实体

继续沿用并升级这些实体：

- `task`
- `task_reward_plan`
- `submission`
- `submission_reward`
- `ledger_account`
- `ledger_entry`

关键约束：

- 任务、投稿、奖励、账务都必须挂到真实用户 ID
- 激活角色属于 session，不属于 user 静态字段
- operator 动作不能直接覆盖业务事实，必须保留审计轨迹

## 10. 契约与状态模型

`packages/contracts` 需要从现有 route blueprint 继续扩展，至少补齐：

- 三端导航口径
- 角色矩阵
- 页面读模型类型
- API 输入输出类型
- 状态枚举

统一状态要点：

- `task` 状态与 `ledger` 状态分离
- `submission` 审核状态与 `reward` 状态分离
- `user` 可用状态与 `session` 当前激活角色分离
- `task governance state` 只影响平台可见性，不直接替代业务阶段状态

## 12. 实施顺序

为了加快交付速度，实施顺序不按应用拆，而按“统一底座 -> 三端接入”推进。

建议顺序：

1. 数据库与 API 认证、会话、角色切换
2. API 的 admin 路由与 operator 动作
3. Web 应用壳、登录和双角色导航
4. Web 创作者与商家主链接入
5. 小程序从 demo session 切到真实 session
6. Admin 从 preview 数据切到真实 API

这样做的原因是：

- 小程序已经有页面，不应成为重写项目
- 统一底座先成型后，Web、Mini Program、Admin 的增量都能直接落到真数据上
- 可以避免先做 Web 演示壳再返工 API

## 13. 成功标准

本轮交付完成后，应满足以下结果：

- Web 与小程序都能登录，并基于真实后端恢复会话
- 双角色用户可在 Web 与小程序中切换商家/创作者身份
- 商家可在 Web 与小程序完成任务创建、审核、评奖、结算
- 创作者可在 Web 与小程序完成浏览、投稿、收益查看
- Admin 可查看真实任务、用户、账务数据并执行基础治理动作
- API 不再依赖 `x-demo-user` 作为真实访问控制方案
