# 架构规划

## 目标

这个 monorepo 不是先选技术再找业务，而是直接围绕功能文档中的三条闭环组织：

- 商家侧：入驻、托管、发任务、审稿、评奖、复盘。
- 创作者侧：注册、找任务、投稿、收益解冻、提现。
- 平台侧：审核、风控、申诉、资金、系统治理。

因此，基础架构采用“应用壳 + 领域包 + 契约层 + harness”的结构，而不是一开始就把所有业务揉进单个前后端目录。

## Monorepo 分层

### `apps`

- `apps/app`
  面向移动 App 与小程序的统一业务壳。这里先承载信息架构、导航分区和跨角色共享能力，后续可按团队选择接入 `Taro`、`uni-app` 或 React Native。
- `apps/admin`
  面向运营和官方后台的 PC 管理台壳。这里优先沉淀页面分区、权限边界和风控/资金工作台，而不是先做 UI 细节。
- `apps/harness`
  回归场景执行器。后续所有 AI 生成页面、API 或工作流变更，都要能映射回这里的场景定义。

### `packages`

- `domain-core`
  workspace manifest、流程定义、harness 类型。
- `domain-user`
  用户、身份、信用、消息。
- `domain-task`
  任务、投稿、评审、效果复盘。
- `domain-finance`
  托管、结算、退款、提现、抽成、对账。
- `domain-risk`
  AI 风控、违规、申诉、裁决、信用联动。
- `contracts`
  三端路由、权限与后续 API 契约的统一入口。

## 为什么这样拆

### 1. 任务和资金必须解耦但强绑定

发布任务前必须完成预算托管，这是平台可信性的底层约束。`task` 负责定义业务动作，`finance` 负责资金状态机，两者通过流程契约绑定，而不是互相吞掉边界。

### 2. 风控和申诉不能只是后台页面

这个产品的风险不在“有无审核页面”，而在“审核结果是否真正回写信用、权限和账务”。因此把 `risk` 单独作为领域包和后台工作台，而不是后台里的一个松散菜单。

### 3. harness 必须单独成应用

如果把 harness 写成零散测试文件，vibe coding 过程中很容易被绕过。单独成 `apps/harness` 后，它有自己的脚本、依赖和入口，天然适合作为 CI 门禁和迭代验收层。

## 推荐后续技术落位

- 用户端：优先 `Taro + React` 或 `uni-app x`，原因是 App/小程序双端同步要求明确。
- 后台端：`Next.js` 或 `Vite + React` 即可，关键是路由和权限体系先稳定。
- BFF/API：建议单独新增 `apps/api`，以 `NestJS` 或 `Hono` 承接工作流编排。
- 数据层：建议单独新增 `packages/database`，统一 schema、迁移与审计字段。
- 异步任务：结算、通知、风控扫描建议单独新增 `packages/workflows`。

## 数据与状态主线

- 用户身份与信用是横切能力，不能分散在任务和后台各自维护。
- 任务状态与资金状态必须可追溯，所有关键节点都要留存操作人、时间和原因。
- 申诉裁决属于跨域事件，至少要同时影响风控域、账务域和通知域。
- 后台所有高风险动作都要进入审计日志。
